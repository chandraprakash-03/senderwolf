# Senderwolf Code Review (April 28, 2026)

## Scope

This review covered the full public API surface and core transport pipeline:

- Entry points and exports (`index.js`, `mod.js`)
- Core delivery orchestration (`lib/sendEmail.js`)
- SMTP transport and MIME assembly (`lib/smtpClient.js`)
- Connection pooling (`lib/connectionPool.js`)
- Validation and config loading (`lib/schema.js`, `lib/config.js`)
- Supporting modules (retry, events, providers, templates)

## High-priority findings

### 1) Pool key collisions caused connection loss and stale resource tracking

**Impact:** Reliability/performance bug under concurrency.

The previous pool used `Map<key, connection>` with one connection per key. When a second concurrent connection was created for the same key, the map entry was overwritten, which could orphan live connections and break accurate lifecycle tracking.

**Fix implemented:**

- Switched pool storage to `Map<key, Set<connection>>`.
- Reuse logic now scans the set for idle/non-expired connections.
- Close handlers now remove the specific connection from the set and clean up empty buckets.

### 2) Queue draining logic did not service queued work from idle connections

**Impact:** Throughput stalls under load at max pool size.

When the queue filled (because active connections hit max), `processQueue()` tried to create new connections directly and returned early when already at max. Idle connections were available but queued jobs were not promptly reassigned.

**Fix implemented:**

- `processQueue()` now replays queued requests back through `getConnection()`, allowing idle connections to satisfy waiting jobs before attempting new connections.

### 3) `activeConnections` bookkeeping could go negative on failed connect attempts

**Impact:** Incorrect pool stats and behavior gating on max connection count.

The previous implementation decremented `activeConnections` in a `catch` path even though incrementing occurred only after successful connect.

**Fix implemented:**

- Removed the incorrect failure-path decrement.
- Added bounded decrement (`Math.max(0, ...)`) in connection close handler as additional guard.

### 4) SMTP envelope sender could include a formatted display name

**Impact:** SMTP protocol correctness; potential delivery failures.

`sendEmail()` built envelope sender from `mail.from || mail.fromEmail`. If `mail.from` was formatted as `Name <email@domain>`, the SMTP client emitted `MAIL FROM:<Name <email@domain>>`, which is invalid.

**Fix implemented:**

- Envelope sender is now always `mail.fromEmail`.
- Display-name formatting remains in the `From:` header (`fromHeader`).

## Medium-priority findings (recommended follow-up)

1. **STARTTLS policy nuance**: client upgrades only when `requireTLS` is true; a mode that prefers STARTTLS when offered (without requiring it) could improve security defaults.
2. **Header injection hardening**: mail header fields should defensively reject CRLF in user-provided values.
3. **Memory pressure for large attachments**: attachments are fully read into memory; streaming support would improve large payload handling.
4. **Potentially misleading stats**: `messagesSent` is exposed in pool stats but currently not incremented.

## Validation checks run

- Module syntax checks via Node import execution.
- Git diff inspection for changed files only.

## Summary

The implemented fixes address correctness issues in two critical paths: **connection pooling concurrency** and **SMTP envelope compliance**. These are low-risk changes with strong reliability benefit.
